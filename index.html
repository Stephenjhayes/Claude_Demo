🔧 Phase 1 — Installation & First Run
01 ✓
Install Claude Code
npm / homebrew / winget · requires Node 18+
Setup
⌄
💡

Claude Code is a CLI tool that lives in your terminal. It reads your entire codebase and executes tasks using natural language. Install it globally so it's available in every project.

Choose your install method

macOS — Homebrew (recommended, auto-updates)
brew install claude-code
Windows — WinGet
winget install Anthropic.ClaudeCode
npm (all platforms)
npm install -g @anthropic-ai/claude-code

Verify install

terminal
claude --version
# → Claude Code v1.x.x

claude doctor
# → Runs diagnostics, checks login status
💡 Windows users: Install Git for Windows first. Windows ARM64 requires a remote session (WSL2 or SSH).
02 ✓
Start your first session
claude · login · explore the TUI
Setup
⌄
🚀

Navigate to your project directory and run claude. On first launch you'll be prompted to log in. After that you're in an interactive session with full access to your codebase.

terminal
cd /your/project

claude
# → Opens TUI, prompts login on first use
# → After login, you're live

Essential first commands to try

  • /help — see all available slash commands
  • /status — show model, permissions, and session info
  • Type: "Explain the structure of this project"
  • Try: "List all TODO comments in the codebase"
  • Ctrl+R — search prompt history
💡 Tip: Use claude -p "your task" for non-interactive / scripted use (great for CI pipelines).
📋 Phase 2 — Project Configuration
03 ✓
Create your CLAUDE.md
Project memory · coding standards · architecture
Config
⌄
🧠

CLAUDE.md is a markdown file in your project root that Claude Code reads at the start of every session. Use it to encode your team's coding standards, architecture decisions, preferred libraries, and review checklists — so you never re-explain them.

Minimal starter template

CLAUDE.md
# My App

## Tech Stack
- Frontend: React + TypeScript
- Backend: Node.js / Express
- DB: PostgreSQL

## Coding Standards
- Functional components only, no class components
- All async functions must handle errors explicitly
- Run `npm run lint` after every TypeScript edit

## Common Commands
- `npm run dev`   → start dev server
- `npm run test`  → run tests

## Notes for Claude Code
- DB migrations go in src/db/migrations/ with timestamp
- Never commit .env directly — use .env.example
📁 See the full template: examples/CLAUDE.md.template in this repo — copy it into your project and customize.
04 ✓
Configure permissions
.claude/settings.json · allow / deny rules
Config
⌄
🔐

Claude Code asks permission before running bash commands. You can pre-approve common safe operations (like npm run * and git *) and pre-deny dangerous ones. Settings can be per-project or global.

Project-level settings

.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(prettier *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(sudo *)"
    ]
  }
}

Manage interactively

inside claude session
/permissions
# → Opens searchable permission manager
# → Use / to filter by tool name
💡 Tip: Commit .claude/settings.json to your repo so every team member starts with the same safe defaults.
⚡ Phase 3 — Power Features
05 ✓
Create custom slash commands
.claude/commands/ · reusable team workflows
Power
⌄
⚡

Slash commands let you package repeatable workflows your team can share. Create a markdown file in .claude/commands/ and it becomes a /command-name you can invoke instantly.

Create a command

.claude/commands/review-pr.md
# Review Pull Request

Perform a code review of the current branch vs main.

## Steps
1. Run `git diff main...HEAD`
2. Check for bugs, security issues, missing tests
3. Run `npm run lint` and `npm run test`

## Output Format
- **Summary**: overview of changes
- **Issues**: file:line + severity
- **Recommendation**: Approve / Request Changes

Use your command

inside claude session
/review-pr
# → Claude runs your custom workflow

/deploy-staging
# → Another example — see examples/.claude/commands/
📁 Ready-made examples: examples/.claude/commands/review-pr.md and examples/.claude/commands/deploy-staging.md
06 ✓
Set up hooks
PreToolUse / PostToolUse · auto-format, safety checks
Power
⌄
🪝

Hooks run shell commands before or after Claude Code actions. Use them to auto-format every file Claude edits, run lint before commits, or block dangerous operations before they happen.

Hook lifecycle

event sequence
PreToolUse  →  Claude runs tool  →  PostToolUse
    ↑                                     ↑
 Safety check                       Auto-format / log

Wire up auto-formatting

.claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit|MultiEdit",
      "hooks": [{
        "type": "command",
        "command": "prettier --write $CLAUDE_FILE_PATH"
      }]
    }]
  }
}
📁 Full example hook script: examples/hooks/post-edit-format.sh — handles TS, Python, Go, and JSON automatically.
07 ✓
Connect MCP servers
GitHub · Slack · Jira · custom integrations
Power
⌄
🔌

The Model Context Protocol (MCP) connects Claude Code to external data sources. With it, Claude can read your Jira tickets, push to GitHub, message in Slack, or query your database — all from inside the session.

Project MCP config

.mcp.json (in project root)
{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

Add via CLI

terminal
# Add GitHub MCP server to project
claude mcp add github \
  --command "npx -y @modelcontextprotocol/server-github" \
  --env "GITHUB_PERSONAL_ACCESS_TOKEN=your_token"

# Then inside Claude: "Create a PR for my current branch"
📁 Full MCP config with 4 servers: examples/.mcp.json.example — covers GitHub, Slack, filesystem, and PostgreSQL.
🖥️ Phase 3b — IDE Integration
VS ✓
Install the VS Code extension
Inline diffs · plan preview · @-mentions · conversation history
Config
⌄
🖥️

The VS Code extension adds an IDE-native layer on top of the terminal experience: you can review Claude's file changes as inline diffs before they're applied, comment on Claude's plan before it executes, and reference files with @-mentions. Conversation history also persists across sessions.

Install

VS Code — search Extensions
# Option A: from Extensions panel (Cmd+Shift+X)
Search: "Claude Code"  →  Install

# Option B: command line
code --install-extension anthropic.claude-code

# Cursor users — same extension works
cursor --install-extension anthropic.claude-code

Open the panel

VS Code command palette
# Cmd+Shift+P → type:
Claude Code: Open in New Tab

# Or use the shortcut:
Cmd+Shift+C  (Mac)  /  Ctrl+Shift+C  (Win/Linux)

What's different vs terminal

  • Inline diffs — see every change before it lands, accept/reject per hunk
  • Plan preview — comment on Claude's plan mid-execution to course-correct
  • @-mentions — @Button.tsx or #isLoading for precise context
  • Persistent history — conversations survive session restarts
💡 Pro move: Dock the panel to your right sidebar and use plan preview before any large refactor. Ask Claude to "make a plan first" — review, comment, then approve.
🤖 Phase 4 — Multi-Agent & Automation
08 ✓
Orchestrate sub-agents
.claude/agents/ · parallel work · checkpoints
Advanced
⌄
🤖

Sub-agents let you spawn multiple Claude instances working in parallel. A lead agent coordinates the work — one agent builds the frontend, another builds the backend API, a third writes tests. Checkpoints let you rewind if anything goes wrong.

Define a specialized agent

.claude/agents/frontend-agent.md
# Frontend Agent

## Role
Specialized in React components and UI logic.

## Responsibilities
- Build components in src/components/
- Handle client-side state
- Run npm run test -- --testPathPattern=components

## Constraints
- Do NOT touch src/api/ — backend agent owns that

## Handoff Signal
End response with FRONTEND COMPLETE when done

Trigger from a session

prompt to Claude Code
# Ask Claude to spawn sub-agents:
"Build the full user auth feature. Use frontend-agent
to build the login/signup forms and backend-agent to 
build the /auth API endpoints. Run in parallel."

# Checkpoints — auto-saved before each change
# Rewind with: Esc Esc  or  /rewind
  • Create .claude/agents/ directory in your project
  • Write an agent markdown file for each domain (frontend, backend, testing)
  • Try a prompt that asks for parallel work on two features
  • Practice using /rewind to roll back a change
📁 See the agent template: examples/.claude/agents/frontend-agent.md — copy and adapt per domain.
